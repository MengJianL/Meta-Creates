## DreamMeta 是什么？

DreamMeta 是一个**元部门（Meta-Department）**——一种受真实企业组织结构启发的 AI 智能体系统组织模式。它不依赖单体提示词或临时编排的 Agent 链路，而是将智能体协同分解为 **13 个不可约原子单元**，排列在三层引力结构中。

元部门是**「创造工具的工具」**。它的核心产出是**项目 Agent 定义**，而非工作成果物——先造工具，再用工具做事。

### 为什么选择 DreamMeta？

- **治理优先** — 每个原子都回答：我是什么、我不是什么、我如何被治理、我如何进化
- **纯抽象** — 不含任何领域专属术语，可直接移植到任意项目
- **自进化** — 每一层都内置反馈回路、伤疤协议（Scar Protocol）与进化落盘机制
- **三权分立** — 执行、核验、评估在结构上强制隔离
- **五级创造** — 从项目 Agent（L0）到元部门完整复制（L4）

> [!NOTE]
> 这不是一个传统的软件项目（没有构建/测试命令）。它是一个**配置即架构（Configuration-as-Architecture）**系统——`.md` 文件定义 Agent 行为，结构本身就是产品。

## 快速开始

### 1. 引导新项目

```bash
# 将 13 原子 + 架构索引复制到你的项目
cp -r .claude/agents/ your-project/.claude/agents/
cp -r .claude/commands/ your-project/.claude/commands/
```

### 2. 运行元部门

在 [Claude Code](https://claude.ai/code) 中使用 `/meta` 斜杠命令：

```
/meta <在此描述你的任务>
```

元部门将依次执行：
1. **意图放大** — 沿三个维度拓展模糊任务
2. **分析拆解** — 结构化分解任务
3. **创建或复用**项目 Agent 定义
4. **派遣执行** — 通过 Agent 定义执行任务
5. **独立核验与评估** — 三权分立
6. **综合交付** — 输出最终结果

### 3. MCP 服务器（可选）

```bash
npm install
npm run mcp:start
```

通过 MCP 协议暴露 `list_atoms` 和 `get_atom` 工具，支持程序化访问 13 原子架构。

## 架构

### 三层引力结构

```
┌─────────────────────────────────────────────────────────────────────┐
│  第三层 — 执行层（Execution）                                        │
│  M09 构成 · M10 检索 · M11 调用 · M12 验证 · M13 创造                │
├─────────────────────────────────────────────────────────────────────┤
│  第二层 — 编排层（Orchestration）                                     │
│  M04 分解 · M05 路由 · M06 评估 · M07 聚合 · M08 序列                │
├─────────────────────────────────────────────────────────────────────┤
│  第一层 — 基础层（Foundation）                                        │
│  M01 记忆 · M02 身份 · M03 通信                                      │
└─────────────────────────────────────────────────────────────────────┘
```

| 层级 | 职责 | 原子 |
|------|------|------|
| **基础层** | 存储、身份、通信 | M01 记忆 · M02 身份 · M03 通信 |
| **编排层** | 协调、路由、评估、排序 | M04 分解 · M05 路由 · M06 评估 · M07 聚合 · M08 序列 |
| **执行层** | 原子操作 | M09 构成 · M10 检索 · M11 调用 · M12 验证 · M13 创造 |

### 三层工具层级

```
┌────────────────────────────────────────────────┐
│  项目 Agent 定义（agents/）                      │  ← 由元部门创建
├────────────────────────────────────────────────┤
│  13 原子（.claude/agents/）                      │  ← 创建 Agent 的基础设施
├────────────────────────────────────────────────┤
│  Claude Code Agent 工具                          │  ← 运行时执行
└────────────────────────────────────────────────┘
```

## 13 原子

| 编号 | 原子 | 层级 | 核心职责 |
|------|------|------|----------|
| M01 | **Memory** 记忆元 | 基础层 | 分层存储、压缩、伤疤协议 |
| M02 | **Identity** 身份元 | 基础层 | 角色边界、行为约束、能力画像 |
| M03 | **Channel** 通信元 | 基础层 | 层间信息流转与过滤 |
| M04 | **Decompose** 分解元 | 编排层 | 意图放大与任务拆解 |
| M05 | **Route** 路由元 | 编排层 | 能力匹配路由、复用优先 |
| M06 | **Evaluate** 评估元 | 编排层 | 多维质量评估、元评审 |
| M07 | **Synthesize** 聚合元 | 编排层 | 多源整合与冲突消解 |
| M08 | **Sequence** 序列元 | 编排层 | 流水线编排与注意力成本管理 |
| M09 | **Compose** 构成元 | 执行层 | 在约束下构成候选内容 |
| M10 | **Retrieve** 检索元 | 执行层 | 定位并提取上下文相关信息 |
| M11 | **Invoke** 调用元 | 执行层 | 触发外部工具/API 并处理结果 |
| M12 | **Verify** 验证元 | 执行层 | 约束核验、反套话检测、门控裁决 |
| M13 | **Create** 创造元 | 执行层 | 模式重组、Agent 创造（L0–L4） |

### 五条核心链路

```
标准任务:     M03 → M04 → M05 → M08 → M09/M10/M11 → M12 → M06 → M07 → M03
候选生成:     M04 → M05 → M09 → M12 → M06 → M07
检索复用:     M04/M05 → M10 → M12/M06 → M07
外部能力:     M05 → M11 → M12 → M06 → M07
能力创造:     M05/M10/M01 → M13 → M12 → M06 → M01 → M05
```

### 两条治理闭环

- **质量闭环**：执行 → M12 核验 → M06 评估 → 回流 → 重新执行
- **进化闭环**：历史缺口 → M13 创造 → M12 核验 → M06 评估 → M01 持久化 → M05 路由

## 项目 Agent 体系

每个项目 Agent 均为**三位一体武装**（Prompt + Skill + Tools），遵循 **9 段模板**：

| 段落 | 用途 |
|------|------|
| **身份定位** | 角色、领域专长、决策权限 |
| **原子映射** | 实例化的 M## 原子 |
| **输入契约** | 接受的格式、验证规则、前置条件 |
| **输出契约** | 正常产出、失败报告、下游 |
| **执行规程** | 逐步操作流程 |
| **行为约束** | 必须做、禁止做、上下文隔离 |
| **质量标准** | 量化验收条件 |
| **技能装备** | 核心 + 辅助技能（自动匹配） |
| **工具优先级** | 主力、辅助、禁用工具（自动匹配） |

### 路由优先级（M05）

1. **命中 `agents/`** — 复用已有项目 Agent
2. **命中全局资源** — 通过 Skill 工具直接调用
3. **无命中** — 创建新 Agent，核验后执行
4. **部分命中** — 升级 Agent 版本

## M13：五级创造体系

| 级别 | 创造对象 | 门控 |
|------|----------|------|
| **L0** 项目 Agent | 任务特定的 Agent 定义 | 无 — 标准产出 |
| **L1** Skill | 可复用模式与模块 | 双重验证 |
| **L2** Agent | 具有身份的新原子单元 | 五门流程 |
| **L3** 元部门 | 完整 13 原子系统实例 | 血统追溯 |
| **L4** 治理感知复制 | 父子元部门体系 | 结构化治理 |

> [!TIP]
> L0 是最常见的创造——每次任务中，元部门首先创建或复用 Agent 定义，再派遣执行。这正是「创造工具的工具」理念的体现。

## 设计原则

1. **不可约性** — 任何原子若进一步分解，将丧失独立的语义完整性
2. **纯抽象** — 不含领域专属术语，可跨领域复用
3. **自进化** — 每个原子内置反馈驱动的自适应机制
4. **正交性** — 边界清晰，职能无重叠
5. **治理优先** — 每个原子定义：我是什么、我不是什么、我如何被治理、我如何进化
6. **三权分立** — M06（评估）和 M12（验证）在结构上被禁止执行写入操作
