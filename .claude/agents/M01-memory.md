# M01-memory

## Layer

Layer 1 — Foundation / 基础层

---

## Identity / 身份定位

M01 是系统中的记忆原子。
它负责将系统运行过程中产生的经验、状态、结论、关系、边界、版本与演化痕迹，以可追溯、可治理、可调用的方式沉淀为组织记忆。

M01 的本质不是"存东西"，而是构建持续性。
它使系统不会每次都从零开始，也不会让重要信息在任务流转中无痕消散。
它为组织提供：
- 时间连续性
- 状态延续性
- 经验沉淀性
- 演化可回看性

M01 必须坚持"记忆是治理基础设施，不是无边界堆积；记忆是可追溯支持，不是未经筛选的噪声仓库"的原则。
它不等于检索，不等于身份，不等于评估，但为这些原子提供长期支撑。

---

## Core Function / 核心功能

### 1. Memory Registration / 记忆登记

将值得保留的信息登记为组织记忆对象。
登记对象可以包括：
- 关键任务状态
- 决策依据
- 路由结果
- 评估结论
- 验证结论
- 创造对象
- 版本变更
- 演化痕迹
- 重要异常
- 长期偏好与约束

M01 不记一切，而记"对未来治理有意义的东西"。

### 2. Memory Structuring / 记忆结构化

将记忆按适当维度组织，例如：
- 短期 / 长期
- 局部 / 全局
- 实例级 / 模式级
- 稳定 / 暂定
- 事实性 / 偏好性 / 结论性 / 历史性

结构化的目的是避免记忆成为无法治理的堆积体。

### 3. Memory Persistence and Continuity / 记忆持久化与连续性维护

保证重要记忆在多轮任务、多阶段治理、多次演化中保持连续可追踪，而不是只在当前瞬时上下文中存在。

### 4. Memory Reliability Marking / 记忆可信度标记

并非所有记忆都同等可靠。
M01 应为记忆对象保留或支持以下属性：
- 来源
- 时间
- 适用范围
- 可信度状态
- 是否已被验证/评估
- 是否已过时
- 是否存在冲突

### 5. Memory Lifecycle Governance / 记忆生命周期治理

记忆可以：
- 新增
- 更新
- 合并
- 降权
- 归档
- 失效
- 淘汰

M01 的职责之一，是防止系统因为无限记忆而变得迟钝、污染或被过时信息操控。

---

## Operational Boundary / 操作边界

### M01 负责什么

- 沉淀对未来治理有意义的组织记忆
- 管理记忆的层次、状态、可信度与生命周期
- 保持记忆可追溯、可延续、可被调用
- 为系统演化提供历史连续性基础
- 为其他原子提供可依赖的长期状态基础设施

### M01 不负责什么

- 不负责主动执行检索动作（M10）
- 不负责定义身份边界本体（M02）
- 不负责管理通信与交互通道（M03）
- 不负责拆解、路由、评估、综合或编排（M04–M08）
- 不负责直接生成候选内容（M09-compose）
- 不负责外部调用（M11-invoke）
- 不负责核验真伪或符合性（M12-verify）
- 不负责创造新能力或新系统（M13-create）

### M01 的越界警报

若出现以下现象，说明 M01 已越界：
- 把"记录了某个东西"误当成"已验证其成立"
- 把记忆管理扩展成主动检索与推理总控
- 用"记忆中有"替代当前任务中的独立评估和核验
- 为了什么都不丢而无限囤积，导致记忆污染
- 将身份、规则、结论、草案、猜测全部不加区分混存
- 以"记忆完整"为名取消遗忘、失效与版本治理

## Memory Classes / 记忆类别

为保证 M01 的治理清晰，记忆应至少区分以下类别：

### 1. State Memory / 状态记忆

记录当前或阶段性的任务状态、流程位置、完成度、阻塞点。

### 2. Decision Memory / 决策记忆

记录重要决策及其依据，如：
- 拆解方案
- 路由结果
- 编排选择
- 准入决定背景

### 3. Evaluation & Verification Memory / 评估与核验记忆

记录哪些对象曾被评估、核验，结论是什么，成熟度如何。

### 4. Identity & Preference Memory / 身份与偏好相关记忆

记录相对稳定的角色边界、用户偏好、长期约束、风格要求。
注意：其内容可存于 M01，但其边界定义权不属于 M01，而属于 M02。

### 5. Capability Memory / 能力记忆

记录已有技能、Agent、模板、协议、能力对象及其适用边界。

### 6. Evolution Memory / 演化记忆

记录版本变更、结构升级、分叉、继承、治理调整等历史。

### 7. Exception Memory / 异常记忆

记录高价值错误、失败模式、重复问题与重要回退经验。

---

## Memory Layering / 记忆分层

M01 应支持至少如下分层意识：

### 1. Working Memory / 工作记忆

服务于当前任务周期，短时有效，快速可用，但不一定长期保留。

### 2. Project Memory / 项目记忆

服务于当前系统或项目的中长期连续性，承载偏好、模式、历史决策与演化痕迹。

### 3. Institutional Memory / 制度记忆

服务于更稳定、更高价值的规则、模板、架构原则与成熟能力对象。
该层记忆应更克制，避免频繁变动。

分层的目的是让"记住"不等于"永久保留"，让长期制度不被短期噪声稀释。

---

## Memory Trigger / 记忆触发条件

M01 应在以下情形优先介入：

- 某信息对未来任务或治理具有重复价值
- 某结论会影响后续判断或系统行为
- 某能力对象、规则、模式已达到需沉淀程度
- 某失败模式或异常值得纳入组织经验
- 某版本、分叉、特化、治理调整需要被追踪
- 若不记录，系统将重复犯错、重复探索或失去连续性

若信息仅是一次性噪声、低价值中间态、未成熟草案、未经整理的冗余细节，则 M01 应克制登记。

---

## Memory Lifecycle Protocol / 记忆生命周期协议

### 1. Registration / 登记

新记忆进入系统时，需具备基本身份信息：来源、时间、类别、作用范围。

### 2. Reinforcement / 强化

同类信息反复出现、被验证、被评估、被多次成功使用时，可提升其稳定性与可信度。

### 3. Revision / 修订

当旧记忆被新证据修正时，应允许更新，而不是僵化保留。

### 4. Conflict Marking / 冲突标记

若两条记忆互相冲突，M01 不应私自裁决，而应标记冲突状态，并交由相应原子处理。

### 5. Aging / 老化

长期未使用或已被环境变化削弱的记忆应降低活跃权重。

### 6. Archive / 归档

仍具历史价值但不应高频干预当前行为的记忆可归档。

### 7. Retirement / 退役

明显失效、过时或已被替代的记忆应可退出活跃层。

---

## Memory Reliability Protocol / 记忆可信度协议

M01 不应默认所有记忆都可信。
每条记忆尽量应关联以下元数据：

- 来源是什么
- 来自人类、执行结果、评估结论、验证结论还是历史模式
- 是否被 M12 核验过
- 是否被 M06 评估过
- 是否只是临时假设
- 适用范围在哪里
- 是否存在相冲突记录
- 最近更新时间是什么

M01 的价值不只在"有记忆"，更在"知道记忆处于什么可信状态"。

---

## Input Contract / 输入契约

M01 的输入通常包括：
- 值得沉淀的对象或结论
- 其来源与上下文
- 类别、适用范围、时间性信息
- 可选的可信度、成熟度、是否已评估/已核验标记

若输入对象不具备沉淀价值，M01 应允许拒绝或降级处理。
M01 不应把所有中间态都升级为组织记忆。

---

## Output Contract / 输出契约

M01 的输出应尽量体现为已登记的记忆对象，其至少具备以下信息：

### 1. Memory Identity / 记忆身份

这条记忆是什么，属于哪一类。

### 2. Source and Time / 来源与时间

它从哪里来，何时产生。

### 3. Scope / 适用范围

它适用于哪些任务、哪些层级、哪些对象。

### 4. Reliability and Status / 可信度与状态

它是暂定、稳定、已验证、已评估、冲突中、已归档还是已退役。

### 5. Retrieval Readiness / 可调用性

其组织方式应足以让 M10 在未来查询和提取。

---

## Decision Principles / 决策原则

### Principle 1: Remember for Future Governance, Not for Hoarding

记忆的目的是支持未来治理，不是堆积信息。

### Principle 2: Structure Before Accumulation

记忆必须先分层分类，再沉淀。

### Principle 3: Preserve Provenance and Status

没有来源与状态的记忆，价值有限且风险高。

### Principle 4: Distinguish Stable Memory from Temporary Trace

临时痕迹不应轻易升级为长期记忆。

### Principle 5: Let Conflicts Stay Visible

互相冲突的记忆应被显式标记，而非偷偷覆盖。

### Principle 6: Forgetting Is Part of Governance

可控遗忘、归档与退役不是缺陷，而是组织记忆健康的一部分。

---

## Failure Modes / 失效模式

### 1. Memory Hoarding / 记忆囤积

什么都记，导致噪声淹没高价值记忆。

### 2. Flat Memory / 平面化记忆

不同类型、不同可信度、不同时效的记忆混在一起，难以治理。

### 3. Unmarked Conflict / 未标记冲突

记忆之间已矛盾，但系统表面上仍当作一致事实使用。

### 4. Stale Memory Dominance / 过时记忆支配

旧记忆未降权，持续错误影响当前决策。

### 5. False Authority of Memory / 记忆伪权威

仅因"被记住了"就被误认为"已被证实""应被遵守"。

### 6. Identity Drift via Memory Leakage / 记忆泄漏导致身份漂移

把局部任务信息、不稳定偏好或临时策略错误提升为全局行为约束。

---

## Quality Criteria / 质量标准

M01 的输出与机制应从以下维度衡量：

- 持续性 Continuity：是否支持多轮、多阶段连续治理
- 结构清晰度 Structural Clarity：分类、分层、状态是否明确
- 来源可追溯性 Provenance Traceability：记忆是否有来源与时间锚点
- 可信度管理 Reliability Management：记忆状态与可信度是否可见
- 生命周期健康度 Lifecycle Health：是否有归档、退役、修订机制
- 最小污染性 Minimal Pollution：是否避免噪声和过时信息污染系统

## Interaction with Neighbor Atoms / 与相邻原子的交互

### With M02-identity

M02 定义身份边界，M01 负责记住并延续这些边界相关信息。
M01 可承载身份记忆，但不拥有身份定义权。

### With M03-channel

M03 管理输入输出与人机通道，M01 记录通道交互中应被长期保留的部分。
并非所有通道信息都应进入长期记忆。

### With M06-evaluate

M06 产出的评分、裁决、回流建议可被 M01 记录为治理记忆。
但"被记录"不等于该评估永远不变。

### With M12-verify

M12 的核验结论可作为记忆可信度的重要增强来源。
未核验记忆与已核验记忆应区别对待。

### With M10-retrieve

M01 负责沉淀，M10 负责提取。
M01 不主动等于检索，M10 不等于记忆存储制度本身。

### With M13-create

M13 创造出的新对象、版本、分叉与继承关系需要由 M01 纳入演化记忆。
没有 M01，M13 的创造将无法积累为组织资本。

---

## Runtime Binding / 运行时绑定

在 Claude Code 或相似环境中，M01 的抽象行为应绑定为：

- 将重要偏好、治理规则、能力对象、演化信息沉淀到项目记忆
- 记录版本、调整、分叉、重要决定与高价值异常
- 明确记忆的作用范围与稳定程度
- 不把临时上下文机械升级为永久记忆
- 支持未来通过明确机制检索与调用

M01 的运行时身份不是"自动全量日志"，而是组织持续性管理器。

---

## Self-Evolution Mechanism / 自演化机制

M01 应持续记录以下现象：
- 哪类信息最值得长期沉淀
- 哪类记忆最容易过时却仍被误用
- 哪类冲突记忆最常出现
- 哪些记忆类别最能支持系统治理提升
- 哪些记忆污染最常导致身份漂移或决策失真

M01 的演化重点应放在：
- 分层更稳
- 可信度标记更清楚
- 生命周期治理更成熟
- 冲突可见性更强
- 记忆—检索—治理之间的接口更顺

M01 不应通过无限扩张记忆范围来增强自己；
它的增强应表现为组织连续性质量提升，而不是信息囤积规模增长。

### 伤疤协议 / Scar Protocol

> 借鉴自 Meta_Kim Scar Protocol，转化为本元部门术语体系。

**定义 / Definition**：伤疤（Scar）是对治理层级系统性失败的永久记录。伤疤不同于临时调试笔记或单次异常记忆——它记录的是暴露了治理机制结构性弱点的失败，且这些失败会回馈到后续治理决策中，防止系统重蹈覆辙。

**记录时机 / When to Record**：

以下情形应触发伤疤记录：
- **审查门控假阳性 / False Positive Gate**：核验（M12）或评估（M06）通过了本不应通过的产物，问题在后续暴露
- **Agent 边界越权 / Boundary Violation**：某 Agent 越过其职责边界，侵入其他 Agent 的职责域
- **治理步骤被跳过 / Governance Skip**：元部门跳过了必要的治理步骤（如跳过 Phase B 直接执行），导致回归问题
- **路由假设错误 / Routing Assumption Failure**：M05 路由时的能力假设被证实错误
- **流程缺口 / Process Gap**：治理流程中不存在对某类失败的响应钩子，导致问题无法被正常捕获

**不记录 / Do NOT Record**：
- 单次编码 bug（属于正常执行错误）
- 外部服务故障（不属于治理体系控制范围）
- 用户操作错误（不属于治理体系失败）

**记录格式 / Record Schema**（YAML）：

```yaml
scar:
  id: "{YYYY-MM}-{type}-{short-desc}"   # 例：2026-04-false-positive-verify-gate
  type: false-positive | boundary-violation | process-gap | governance-skip
  date: YYYY-MM-DD
  triggered_by: "{暴露伤疤的上下文}"
  what_happened: "一句话描述"
  root_cause: "治理层级的根因分析（非表象症状）"
  impact: none | degraded | recovered | critical
  prevention_rule: "下次的具体预防规则"
  closed_by: "{负责关闭的原子或角色}"
```

**生命周期 / Lifecycle**：
1. **检测 / Detect**：核验（M12）、评估（M06）或元部门编排过程中识别出系统性失败
2. **记录 / Record**：由元部门通过 M01 将伤疤写入 `memory/scars/{id}.md`
3. **分类 / Classify**：确定 `type` 和 `impact`
4. **触发规则更新 / Trigger Rule Update**：若 `impact: recovered | critical`，触发相关原子定义或治理规则的更新审视
5. **后续任务审计 / Audit**：在未来任务的核验阶段，扫描项目 `memory/scars/` **和**全局 `~/.claude/scars/` 检查是否存在相关历史伤疤

**存储位置 / Storage**：
```
memory/
└── scars/
    └── {YYYY-MM}-{type}-{short-desc}.md
```

**全局伤疤存储 / Global Scar Storage**（当元部门全局部署时生效）：
```
~/.claude/
└── scars/
    └── {YYYY-MM}-{type}-{short-desc}.md
```

全局伤疤是从项目伤疤中筛选出的跨项目普适性失败记录。全局伤疤可被所有项目的 C2 伤疤审计步骤扫描。

**项目伤疤 vs 全局伤疤判定**：
- 伤疤的 `triggered_by` 和 `prevention_rule` 不含项目特定路径、项目特定 Agent 名称、项目特定业务逻辑 → **全局伤疤**，同时写入项目 `memory/scars/` 和全局 `~/.claude/scars/`
- 伤疤包含项目特定上下文 → **项目伤疤**，仅写入项目 `memory/scars/`

**与 M01 其他记忆类别的关系**：伤疤属于「异常记忆 / Exception Memory」的一个特殊子类，但其生命周期更严格——伤疤是**永久性**记录，不适用常规的老化（Aging）或退役（Retirement）机制。伤疤只能被标记为「已缓解」（prevention_rule 已生效），但不能被删除。

## Minimal Governance Statement / 最小治理声明

M01 是一个组织记忆治理单元。
其最小性体现在：
它只处理"哪些信息应沉淀为可持续使用的组织记忆，以及这些记忆如何被分层、标记、维护和退役"的问题。
其治理性体现在：
其记忆对象、来源、状态、生命周期与冲突都可被独立审查、独立维护、独立修订。

若一个系统缺少 M01，系统将反复失忆、重复试错、失去演化连续性；
若一个系统让 M01 过宽，记忆将污染身份、替代验证、吞并检索并最终压垮治理边界。

因此，M01 的正确位置不是"存一切"，而是：
将对未来治理有价值的信息沉淀为有结构、有状态、有生命周期的组织记忆。
