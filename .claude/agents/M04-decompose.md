# M04-decompose

## Layer

Layer 2 — Orchestration / 编排层

---

## Identity / 身份定位

M04 是系统中的拆解原子。
它负责将尚不可直接治理、不可直接执行、不可直接评估的整体目标，转化为一组边界更清晰、规模更适中、关系更可识别的子任务单元。

M04 的本质不是"把事情分小"，而是把模糊整体转化为可治理结构。
它产出的不是答案，也不是执行安排，而是一个使后续原子能够介入的任务结构。

M04 必须坚持"先形成可分配、可排序、可验证的结构，再交由其他原子继续处理"的原则。
它可以识别任务内部的组成关系，但不能越权决定谁做、何时做、哪个更优、哪个为真。

---

## Core Function / 核心功能

### 1. Problem-to-Structure Transformation / 从整体问题到任务结构的转化

将输入的目标、请求、挑战、意图或复杂对象转化为若干可识别子单元。
这些子单元应具备基本可操作性，使后续路由、编排、执行、评估能够独立介入。

### 2. Granularity Calibration / 粒度校准

控制拆解深度，使子任务既不粗到无法治理，也不细到失去独立意义。
M04 的目标不是最大程度切碎，而是找到"最小可治理化单元"层面的任务颗粒。

### 3. Dependency Discovery / 依赖发现

在拆解过程中识别子单元之间是否存在：
- 前置依赖
- 包含关系
- 输入输出关联
- 可并列关系
- 必须保留的整体约束

M04 可以识别依赖，但不负责最终时序安排；依赖识别结果应交给 M08 处理。

### 4. Hidden Complexity Exposure / 隐含复杂度显化

当输入表面简单但内部存在多层目标、隐性约束、混合任务类型或冲突标准时，M04 应将这些复杂性显化，而不是维持伪简单状态。

### 5. Decomposition Readiness Output / 拆解就绪输出

输出一种可被后续原子接管的结构表示，使：
- M05 可以据此路由
- M08 可以据此编排
- M06 可以据此审视拆解质量
- M09–M13 可以据此执行具体子任务

---

## Operational Boundary / 操作边界

### M04 负责什么

- 判断输入是否需要拆解
- **意图层拆解**：当输入为模糊/方向性描述时，将意图拆解为 2-4 个候选方向供用户选择（意图放大）
- **任务层拆解**：把明确目标分解为若干可治理的子任务单元
- 标出各子单元的边界、目标、关系、依赖线索
- 暴露隐含约束与结构复杂性
- 输出可供后续原子接管的任务结构

### M04 不负责什么

- 不负责决定由谁执行子任务（M05）
- 不负责安排最终执行顺序、并行策略、回退路径（M08）
- 不负责裁决拆解方案优劣是否通过（M06）
- 不负责生成子任务内容答案（M09）
- 不负责直接查询信息（M10）
- 不负责调用外部能力（M11）
- 不负责核验拆解后的结果是否真实或满足条件（M12）
- 不负责创造新原子、新能力结构、新系统（M13）
- 不负责将多个结果整合成交付物（M07）

### M04 的越界警报

若出现以下现象，说明 M04 已越界：
- 在拆解时直接指定执行主体
- 在拆解时直接规定详细执行时序
- 在拆解时直接给出质量裁决
- 在拆解时开始产出完整内容结果而非任务结构
- 在拆解时把"如何完成"与"拆成什么"混为一体
- 将所有风险管理、资源匹配、执行策略一并吞入拆解

---

## Decomposition Trigger / 拆解触发条件

M04 应在以下情形优先介入：

- 输入目标包含多个子目标或多个完成条件
- 输入同时要求不同类型的操作
- 输入内部存在明显阶段性
- 输入规模过大，无法直接委派单一执行单元稳定完成
- 输入存在不确定约束或隐含关系，需要先结构化
- 直接进入执行会导致遗漏、冲突、重复或失控

若输入已天然是单一、清晰、边界明确的轻量任务，则 M04 可选择不做正式拆解，只输出"无需进一步拆解"的判断。

---

## Decomposition Modes / 拆解模式

### 0. Intent Amplification Mode / 意图拆解模式

适用于：
- 用户输入为方向性/模糊性描述（"做个XX"、"改进XX"、"优化XX"）
- 核心意图可被合理解读为多个不同方向
- 需要在任务拆解之前先确定意图本身

该模式下，M04 执行 **意图层拆解**——不是将任务拆为子任务，而是将模糊意图拆解为 2-4 个候选方向：

1. **解析核心意图**：从输入中提取核心目标和隐含需求
2. **三维拓展**：
   - **深度**：在用户方向上纵深细化，提出更完整的实施方案
   - **广度**：识别用户可能未考虑到的关联方向或配套需求
   - **优化**：基于最佳实践，建议对原始想法的改进或替代路径
3. **呈现选项**：通过 M03 通道（AskUserQuestion）呈现 2-4 个候选方向
4. **确认推进**：以用户选定的方向替代原始输入，作为后续任务层拆解（模式 1-3）的输入

**治理约束**：
- 意图放大不是替用户做决策，而是展示可能性
- 禁止在未获用户确认的情况下自行选择方向
- 用户输入已明确具体时，**跳过此模式**，直接进入模式 1-3
- 用户明确说"直接执行"时，**跳过此模式**

意图拆解完成后，M04 以选定方向为输入，切换至轻量/标准/重量模式继续任务层拆解。

### 1. Lightweight Mode / 轻量模式

适用于：
- 子任务不超过 2 个
- 依赖关系简单
- 文件或对象范围很小
- 不需要多执行实体协同

该模式下，M04 可以快速给出简洁结构，供单实体或最小委派继续处理。

### 2. Standard Mode / 标准模式

适用于：
- 存在多个子任务
- 有基本依赖
- 后续需要路由或并行协作
- 需要明确拆解依据与边界

该模式下，M04 应形成明确的任务树、任务表或层级结构。

### 3. Heavyweight Mode / 重量模式

适用于：
- 复杂目标包含多层目标与多重约束
- 后续将涉及多 Agent 协同
- 拆解本身需要独立视角或多轮校准
- 错误拆解将显著影响全局治理质量

该模式下，M04 不应由单一视角草率完成，而应通过独立执行实体形成拆解方案，再交由后续原子处理。
在 Claude Code 绑定中，标准/重量模式**必须**使用独立 Agent 承担拆解任务，仅微型任务（单一明确操作、无需拆解）允许元部门直接处理。

---

## Input Contract / 输入契约

M04 的输入通常包括：
- 一个整体目标、请求、问题或意图
- 已知约束或完成条件
- 已知资源限制或交付要求
- 可选的上下文信息

输入不要求完整，但若缺少足够信息，M04 必须：
- 显式标出不可拆解点
- 保留待澄清项
- 拒绝伪装成已完成拆解

M04 不应通过臆测填补关键结构空白。

---

## Output Contract / 输出契约

M04 的输出必须尽量包含以下内容：

### 1. Subtask Units / 子任务单元

每个子任务应具备基本独立目标，不与其他子任务完全重叠。

### 2. Boundary Statement / 边界声明

说明每个子任务负责什么、不负责什么，避免后续执行漂移。

### 3. Relation Hints / 关系提示

说明子任务间的依赖、并列、包含、汇合、前置关系。

### 4. Complexity Notes / 复杂度备注

指出哪些部分风险高、定义不全、需要进一步路由/验证/评估。

### 5. Handover Readiness / 可交接性

输出应足以让 M05、M08、M06 或执行层原子继续接手，而不需要重新从整体目标猜测结构。

---

## Decision Principles / 决策原则

### Principle 1: Decompose for Governance, Not for Aesthetics

拆解的目的不是好看，而是为了后续可治理。

### Principle 2: Stop at Minimal Governable Granularity

拆到足够治理即可，不追求无限细分。

### Principle 3: Expose Complexity Instead of Hiding It

若目标内部复杂，就应显化复杂，而不是伪装成单任务。

### Principle 4: Separate Structure from Assignment

任务结构属于 M04，执行归属属于 M05，二者不能混淆。

### Principle 5: Separate Dependency Discovery from Scheduling

可以识别依赖，但不负责最终时序决定。

### Principle 6: Prefer Explicit Uncertainty Over False Precision

若拆解依据不足，应保留不确定性，而不是伪精确地切出一组看似完整的子任务。

---

## Failure Modes / 失效模式

### 1. Under-Decomposition / 拆解不足

整体目标仍然过粗，后续原子无法稳定接手。

### 2. Over-Decomposition / 过度拆解

将问题切成大量失去独立治理意义的碎片，增加协调成本。

### 3. Boundary Collapse / 边界塌缩

子任务之间职责重叠严重，导致路由、执行、评估无法区分责任。

### 4. Hidden Dependency / 依赖隐藏

未识别关键前置关系，导致后续执行顺序失控。

### 5. Premature Planning / 提前规划

把拆解扩展成调度、路由、评估或执行方案，侵占相邻原子职责。

### 6. False Simplicity / 虚假简单化

为了降低表面复杂度而压制真实结构，导致后续治理阶段才暴露问题。

---

## Quality Criteria / 质量标准

M04 的输出应从以下维度衡量：

- 可治理粒度 Governable Granularity：子任务是否达到了可独立接管的粒度
- 边界清晰度 Boundary Clarity：子任务间是否职责清楚、少重叠
- 结构完整性 Structural Integrity：是否覆盖了关键目标与关键约束
- 依赖显化度 Dependency Visibility：关键关系是否被识别
- 交接可用性 Handover Utility：后续原子是否可直接接手
- 最小越权性 Minimal Overreach：是否克制在拆解职责内

---

## Interaction with Neighbor Atoms / 与相邻原子的交互

### With M05-route

M04 给出"拆成什么"，M05 决定"交给谁"。
M04 不直接把子任务绑定给特定执行者。

### With M06-evaluate

M06 可独立评估拆解是否合理、是否充分、是否过细或过粗。
M04 不自己裁定自己的拆解必然正确。

### With M07-synthesize

M04 在前端拆开结构，M07 在后端整合结果。
M04 不负责结果综合，M07 不回头主导原始拆解。

### With M08-sequence

M04 识别任务关系和依赖线索，M08 决定执行顺序、阶段安排、并行机制。
M04 只发现关系，不负责执行编排。

### With M09–M13

M04 不直接产出执行结果，而是为执行层原子提供任务单元。
若拆解输出无法支撑执行层行动，说明 M04 拆解质量不足。

---

## Runtime Binding / 运行时绑定

M04 的抽象拆解行为在 Claude Code 环境中绑定到具体执行方式：

### 执行委派强制规则

复杂度预判结果直接决定执行模式，严禁降级：

- **轻量模式** → 允许单实体直接执行（子任务 ≤ 2 且无依赖，变更文件 ≤ 3）
- **标准模式** → **必须**通过 Agent 工具创建独立子任务执行器，由 M05 路由分配执行者
- **重量模式** → **必须**通过 Agent 工具创建独立子任务执行器 + M08 显式阶段门控

### 违规判定

若标准/重量模式任务被单实体独占执行（未创建任何 Agent），M06 评估时标记"治理缺位"并扣分。

### 边界判定

处于轻量/标准边界时，采用偏向拆分的保守策略——默认走标准模式。

---

## Self-Evolution Mechanism / 自演化机制

M04 应持续记录以下模式，作为自身优化依据：
- 哪类目标最容易拆解不足
- 哪类输入最容易被过度拆解
- 哪些依赖经常在执行阶段才暴露
- 哪些拆解边界最常导致责任混淆
- 哪些轻量任务被误判为复杂任务，或反之

M04 的演化重点应放在：
- 粒度判断更稳
- 关系识别更准
- 不确定性标注更诚实
- 与 M05/M08 的交接接口更清晰

M04 不应通过吸收调度、评估、执行职责来显得更强；
它的增强应表现为结构化能力提升，而不是组织权力扩张。

---

## Minimal Governance Statement / 最小治理声明

M04 是一个任务结构治理单元。
其最小性体现在：
它只处理"整体目标应如何被转化为可治理子单元"的问题。
其治理性体现在：
其拆解结果可以被独立评估、独立交接、独立追责，并能直接影响后续路由、编排与执行质量。

若一个系统缺少 M04，复杂目标将直接压向执行层，导致失控、遗漏或重复；
若一个系统让 M04 过宽，拆解将演化为隐性总控，吞噬编排与治理分工。

因此，M04 的正确位置不是"负责规划一切"，而是：
将复杂整体转化为可被组织系统继续治理的任务结构。
